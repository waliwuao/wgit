# wgit 高风险命令安全规范（草案）

本文定义 `wgit` 的高风险操作保护策略，目标是让初学者在执行不可逆或高影响动作前，能够“看得见影响、可中止、可恢复”。

## 风险等级

| 等级 | 定义 | 示例 |
|---|---|---|
| L1 低风险 | 可逆且影响范围小 | 普通切分支、查看状态 |
| L2 中风险 | 可能覆盖工作区状态，但可恢复 | `sync` 自动 stash/rebase |
| L3 高风险 | 可能丢失数据或改写历史 | `undo hard`、冲突中 `reset --hard`、删分支 |

## 通用交互规范

### 1) 先预览，再确认

在执行 L2/L3 操作前，必须展示：

- 将执行的关键 Git 命令摘要（不需要全部参数细节）。
- 影响对象（当前分支、目标分支、待删除分支、目标 commit/reflog）。
- 是否可能丢失未提交内容或改写历史。

### 2) 分级确认策略

- **L2：单次确认**
  - 文案示例：`This may modify local branch state. Continue?`
- **L3：双重确认**
  - 第一次确认：说明风险。
  - 第二次确认：要求输入目标标识（如分支名或 commit hash 末 7 位）以防误触。

### 3) 失败后的最小恢复指引

错误提示必须包含：

1. 当前状态说明（如：rebase in progress）。
2. 立即可执行的 1-3 条命令。
3. 若需中止，明确中止命令。

## 命令级策略

## `undo`

- 风险归类：
  - `soft` 为 L2
  - `hard` 为 L3
- 执行前预览：
  - 目标 commit/reflog 信息。
  - `hard` 模式下明确提示“将丢失未提交改动”。
- 确认策略：
  - `soft`: 单次确认。
  - `hard`: 双重确认 + 输入目标 hash 片段。

## `finish`

- 风险归类：L2/L3（取决于是否触发清理与强制回退）。
- 执行前预览：
  - 源分支、父分支、是否删除源分支、是否创建 tag。
- 冲突场景规范：
  - 不默认触发 `reset --hard`，先提示用户选择。
  - 若用户选“回退”，再展示将执行的恢复动作并二次确认。
- 清理策略：
  - 删除分支前再次确认，提示“是否已推送并完成 PR 合并”。

## `sync`

- 风险归类：L2。
- 执行前预览：
  - 是否检测到未提交改动（是否会 stash）。
  - 目标 upstream 或待选 remote。
- 失败恢复：
  - pull/rebase 失败时，固定展示“冲突解决路径”和“中止路径”。
  - stash 恢复失败时，给出手动恢复建议（`git stash list/apply/pop`）。

## `start`

- 风险归类：L2（当前存在隐式空提交）。
- 执行前预览：
  - 在“无提交历史”时弹窗说明为什么需要基线提交。
- 建议策略：
  - 默认不自动创建空提交，改为确认后执行。
  - 在配置中提供 `auto_bootstrap_commit = true/false`。

## 文案与可读性规则

- 使用短句，避免术语堆叠；错误提示遵循“发生了什么 -> 影响 -> 怎么办”顺序。
- 所有高风险提示统一前缀：`[Safety Check]`。
- 对新手优先显示可执行动作，不先展示底层错误堆栈。

## 验收标准（建议）

1. 所有 L3 操作均具备双重确认。
2. 所有 L2/L3 操作在执行前可看到影响预览。
3. 冲突和失败场景提示可直接复制执行。
4. 新用户可在不查外部资料的前提下完成中止/恢复。
