# wgit 与 GitHub Flow 差距清单

本文从“初学者协作闭环”视角，对 `wgit` 当前能力与标准 GitHub Flow 进行对照，定位缺口与优先级。

## 对照基线（标准 GitHub Flow）

典型流程可抽象为：

1. 从默认分支拉取最新代码。
2. 创建功能分支并开发。
3. 小步提交并推送远程分支。
4. 创建 Pull Request，请求代码评审。
5. 处理评审意见并持续同步分支。
6. PR 合并后清理分支并同步默认分支。

## 当前能力映射

| GitHub Flow 阶段 | `wgit` 现状 | 结论 |
|---|---|---|
| 初始化与默认分支约束 | `init` 可初始化仓库、标准化 `main`、生成 `.git/wgit.toml` | 已覆盖，适合新手 |
| 创建分支 | `start` 提供分支类型选择与命名校验 | 已覆盖，但校验边界不完整 |
| 暂存与提交 | `add` + `commit` 提供交互式选择和结构化提交模板 | 已覆盖，体验良好 |
| 同步远程 | `sync` 支持自动 `stash + pull --rebase + push` | 已覆盖，但失败恢复可读性可加强 |
| 代码评审与合并 | 当前无 `pr` 命令；`finish` 主要是本地 squash merge | 核心缺口（与 GitHub Flow 主路径不一致） |
| 合并后回收与追踪 | `finish` 支持删分支、可选打 tag | 部分覆盖，但偏本地 Git Flow 风格 |

## 关键差距与影响

### P0: 缺少 PR 主路径能力

- 未提供创建 PR、查看 PR 状态、检查合并条件（CI/Review）等操作。
- 对初学者影响：会把“协作评审”退化成“本地合并”，与团队常见 GitHub Flow 规范脱节。

### P1: `finish` 与 GitHub Flow 语义不一致

- `finish` 以本地分支合并为中心，而不是“推送分支 -> 开 PR -> 在平台合并”。
- 对初学者影响：容易形成错误心智模型，误以为本地合并即可完成协作闭环。

### P1: 高风险操作预期不透明

- `start` 在无提交历史时会自动创建空提交。
- `undo` 可执行 `hard reset`，但缺少明确损失预览和分级确认。
- 对初学者影响：出现“工具替我做了决定”的不安全感与误操作风险。

### P2: 规范策略一致性不足

- 配置默认保护分支为 `main`，但多处逻辑又兼容 `master`。
- 分支命名与 tag 校验为局部规则，未形成统一策略文档与一致校验入口。
- 对初学者影响：规则边界不清，遇到失败不知是“Git 限制”还是“工具限制”。

### P2: 工程质量门禁不足

- 当前未见自动化测试矩阵和 CI 质量门禁（`fmt/clippy/test`）。
- 对初学者影响：工具行为不稳定会直接放大学习成本。

## 优先级建议（收益/成本）

| 优先级 | 改进项 | 用户收益 | 实施成本 |
|---|---|---|---|
| P0 | 增加 `pr` 子命令并接入主引导路径 | 高 | 中 |
| P1 | 重构 `finish` 为“本地收尾 + PR 合并后清理”双模式 | 高 | 中 |
| P1 | 高风险操作分级确认与预览 | 高 | 中 |
| P2 | 统一分支与提交规范策略（配置+校验+提示） | 中高 | 中 |
| P2 | 建立最小测试矩阵与 CI 门禁 | 中高 | 中 |

## 建议的短期里程碑

1. 首先补齐 `pr` 能力并把 `menu` 默认流程升级为 GitHub Flow 主链路。
2. 同步上线高风险保护策略，优先覆盖 `undo`、`finish`、`sync`。
3. 将规范策略和新手手册并行发布，降低迁移与认知成本。
